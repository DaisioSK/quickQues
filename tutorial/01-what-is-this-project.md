## 一句话

**给非技术用户用的"PDF 文档智能问答机器人"——领域无关。**

你扔进去一堆扫描的 PDF（合同、财报、规范、手册……都行），它能回答你的中文提问，并告诉你答案来自哪份文件的第几页。

> **定位提醒**：本系统是**领域无关**的文档知识库框架，不是建筑专用。下面用"建筑合同（DEMO）"举例只是因为它是**首个落地领域**（first case study）——把语料换成财报、配一个对应的 DomainProfile，同一套系统照样能用。详见 [`docs/project_guideline.md`](../docs/project_guideline.md) 顶部「定位声明」。

## 一个具体场景（先建立画面）

想象你朋友老王在建筑公司管标书：

- 桌上堆着 9 份 DEMO 项目的 PDF，**总共 4100 页**，大部分是**扫描件**（不能直接复制文字）
- 老王每天被同事追着问：
  - "桥梁防水是谁的责任？"
  - "图纸 T/PRJ/CWD/WS/2101A 在哪份文件里？"
  - "Rev A 跟 Rev 0 的差异有哪些？"
- 老王只能 Ctrl-F 翻 PDF，一翻一下午

**这个项目要做的事**：把这堆 PDF 变成一个**会答题的助手**。老王用中文提问，它用中文回答，**每一句答案都标注来自哪页**，老王点一下就能跳到原文确认。

---

## 它解决什么具体问题

| 老王的痛点 | j-contract 怎么解决 |
|---|---|
| PDF 是扫描件，搜不了文字 | OCR：用 Claude Vision 把图片变成文字（章节 [06](06-pdf-parsing-and-ocr.md)） |
| 文档 4100 页，关键词搜不准 | 双路检索：语义 + 关键词，融合排序（章节 [11](11-rrf-fusion.md)） |
| 不懂英文术语怎么用中文问 | 多语言 embedding：mpnet/bge-m3 同时懂中英（章节 [08](08-embedding.md)） |
| 怕 AI 瞎编 | 强制引用：每句答案必须标 `[文件 p.X]`，假引用自动剥掉（章节 [14](14-answerer-and-citations.md)） |
| 图引条款、Q&A 互引怎么追 | 引用图谱：SQLite 倒排表精确查找（章节 [13](13-ref-graph.md)） |
| 老王不会用命令行 | （Phase 5 计划做 Web UI；目前先用 CLI） |

## 它**不是**什么

为了避免误解，先划清边界：

- ❌ **不是通用 PDF 阅读器** — 它不渲染 PDF，不做翻页，只做问答
- ❌ **不是通用 ChatBot** — 它只回答你给的合同 PDF 里的事，不上网搜，不靠"常识"
- ❌ **不是云服务** — 是你自己机器上跑的代码，PDF 不上传任何第三方
- ❌ **不是商业产品** — 是给一个具体朋友定制的工具，目前 0 用户、0 收入

## 这个项目的"个性"

如果说每个软件项目都有性格，这个项目的关键词是：

1. **极度模块化** — 每个外部依赖都能换。今天 Claude 太贵，明天换 DeepSeek，改一行配置（章节 [04](04-interface-and-impls.md)）
2. **强契约纪律** — 严格按 `dev-contract/` 的规则开发，三件套（ruff + mypy + pytest）门必须绿才能合流
3. **引用至上** — 答案没引用 = 没用。所有架构决策都围绕"能不能溯源到 PDF 某一页"
4. **订阅优先** — 所有 LLM 调用都走 Claude Code 订阅，**不烧 API token**（这是你 2026-05-29 特别要求的）
